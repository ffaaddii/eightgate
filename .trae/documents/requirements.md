## نظام شركة البوابة الثامنة (المرحلة الأولى) — المتطلبات

### 1) الهدف
- توثيق متطلبات تطبيق ويب لإدارة المستخدمين والصلاحيات (RBAC) وإدارة قائمة HSCODE مع سير عمل ناشر/مدقق قبل الإظهار، مع سجل تدقيق وبحث سريع.

### 2) النطاق (Scope)
- إدارة المستخدمين: إنشاء/تعديل/حذف/تعطيل.
- صلاحيات مبنية على الأدوار RBAC على مستوى الشاشات والوظائف (عرض/إضافة/تعديل/حذف/تصدير عند توفره).
- إدارة جدول HSCODE: CRUD.
- سير عمل قبل الإظهار: ناشر يضيف/يعدل ثم مدقق يعتمد/يرفض قبل أن تظهر السجلات لباقي المستخدمين.
- سجل تدقيق: تسجيل دخول/خروج وتغييرات الصلاحيات وعمليات CRUD الحرجة.
- بحث سريع داخل HSCODE.

### 3) المستخدمون والأدوار
- superadmin: إدارة النظام بالكامل.
- customs_broker (الملخص الجمركي): عرض/بحث HSCODE.
- publisher (الناشر): إضافة/تعديل سجلات HSCODE وإرسالها للتدقيق.
- auditor (المدقق): تدقيق السجلات واعتمادها/رفضها.

### 4) متطلبات وظيفية

#### 4.1 المصادقة
- تسجيل الدخول بواسطة اسم مستخدم وكلمة مرور.
- إمكانية تعطيل حساب المستخدم من قبل superadmin.
- تسجيل الخروج.

#### 4.2 إدارة المستخدمين (Users)
- superadmin يستطيع:
  - إنشاء مستخدم جديد وتعيين دوره.
  - تعديل بيانات المستخدم (اسم العرض، الدور، حالة التفعيل).
  - إعادة تعيين كلمة المرور.
  - حذف/أرشفة مستخدم حسب سياسة المرحلة.
- النظام يمنع المستخدم غير المخوّل من الوصول لشاشات/وظائف غير مصرح بها.

#### 4.3 RBAC
- الصلاحيات تُطبق على:
  - مستوى API (حماية المسارات)
  - مستوى الواجهة (إخفاء/تعطيل الإجراءات غير المسموحة)

#### 4.4 سجل التدقيق (Audit Log)
- تسجيل أحداث: الدخول/الخروج، فشل الدخول، إنشاء/تعديل/تعطيل مستخدم، تغيير دور، إنشاء/تعديل/حذف HSCODE، اعتماد/رفض.
- كل سجل يحتوي: الوقت، نوع الحدث، الممثل (actor)، الهدف (target)، وملخص تغييرات.

#### 4.5 إدارة HSCODE + سير العمل
- حالات السجل:
  - pending_review (بانتظار التدقيق)
  - approved (معتمد)
  - rejected (مرفوض)
- الناشر:
  - ينشئ/يعدل سجلات ويجعلها pending_review.
- المدقق:
  - يعتمد السجل لتصبح approved أو يرفضه مع سبب.
- قواعد الإظهار:
  - جميع المستخدمين يرون فقط approved.
  - الناشر يرى سجلاته غير المعتمدة.
  - المدقق يرى جميع السجلات غير المعتمدة.
  - superadmin يرى كل شيء.

#### 4.6 البحث داخل HSCODE
- بحث نصي سريع عبر عدة حقول (مثلاً: code + description + notes).
- دعم التصفية حسب الحالة للمستخدمين المخولين (publisher/auditor/superadmin).

### 5) متطلبات غير وظيفية

#### 5.1 الأمان
- تطبيق سياسة أقل صلاحية (Least Privilege).
- حماية الجلسة/التوكن وعدم تخزين كلمات المرور بنص صريح.
- تسجيل محاولات الدخول الفاشلة.
- تقييد الأجهزة: دعم قائمة أجهزة مسموحة (مستوى MVP يعتمد على معرف جهاز داخل المتصفح، مع توثيق أن المنع الكامل للـ screenshots/video غير ممكن تقنياً من الويب).

#### 5.2 الأداء
- بحث مع Debounce (On-changed/On-unchanged field) لتقليل الاستعلامات.
- فهرسة الحقول الأساسية في HSCODE.

### 6) معايير قبول (Acceptance Criteria)
- مستخدم بدون صلاحية لا يستطيع الوصول لواجهات/مسارات API المحمية.
- لا يظهر أي سجل HSCODE غير معتمد لمستخدم عادي.
- كل عملية اعتماد/رفض تُسجل في Audit Log.
- البحث يُظهر نتائج خلال زمن مقبول مع بيانات تجريبية.
